# Configuration Backup ‚Äî Nilpeter

Small admin app to track hardware configurations (hardware trees) and backups for Nilpeter machines.

## Overview
- Backend: Node.js + Express API that stores hardware trees, nodes and backups in MySQL. Exposes REST endpoints under `/api/*` (equipment, slots, backups, hardware-trees, hardware-nodes).
- Frontend: SvelteKit (Vite) single-page app providing UI for equipment, hardware trees, compare views and import/export.
- Purpose: track hardware configurations across software versions, compare trees to detect added/removed/moved nodes, and manage backups.

## Features
- View and manage hardware trees and their nested nodes (slots).
- Create / copy / import hardware trees from Excel.
- **Rich hardware + IO variable comparison** ‚Äî compare any two software versions of a hardware tree and get a detailed, structured diff of added, removed, modified and moved nodes. Detects physical hardware changes (article number, address, slot) as well as IO variable changes (renamed or removed PLC variable mappings per physical channel). Handles trees with thousands of nodes and IO entries with no noticeable delay.
- Import IO variable mappings from B&R Automation Studio IO exports and attach them as channel-level children of the corresponding hardware modules.
- Upload/download backups associated with equipment.
- Accessible UI improvements (keyboard-friendly modals, labeled inputs, image buttons).

## Repo structure
- `backend/` ‚Äî Express API, DB access, routes.
  - `server.js` ‚Äî app entry, CORS and routes.
  - `.env` ‚Äî runtime settings (DB credentials, PORT, FRONTEND_ORIGIN).
  - `routes/` ‚Äî route handlers for equipment, backups, hardware-trees, etc.
  - `db.js`, `schema.sql` ‚Äî DB connection and schema.
  - `scripts/import-config.js` ‚Äî CLI importer: reads a B&R hardware config Excel export and inserts a hardware tree into the DB.
  - `scripts/import-io.js` ‚Äî CLI importer: reads a B&R IO variable export and attaches variable mappings as children of existing hardware nodes.
  - `scripts/export-template.js` ‚Äî helper to generate example export files.
- `frontend/` ‚Äî SvelteKit app
  - `src/lib/api.js` ‚Äî client API wrapper (reads `VITE_API_URL`).
  - `scripts/sync-backend-port.js` ‚Äî copies backend port into `frontend/.env` as `VITE_API_URL` before build/dev.

## Environment & configuration

Backend `.env` (example):

```
DB_HOST=localhost
DB_PORT=3306
DB_USER=configUser
DB_PASSWORD=secret
DB_NAME=config_backup

PORT=3005
FRONTEND_ORIGIN=http://localhost:5173
```

Notes:
- `PORT` controls what port the backend listens on. The frontend client expects `VITE_API_URL` (set automatically by the sync script).
- `FRONTEND_ORIGIN` is used by the backend CORS policy. In development the server allows any `http://localhost:*` origin.

## Frontend sync script
The frontend includes `scripts/sync-backend-port.js` which reads `backend/.env` for `PORT` (or `BACKEND_PORT`) and writes `frontend/.env` containing `VITE_API_URL=http://localhost:<PORT>`.
The `package.json` `dev`, `build`, and `preview` scripts run this script first so the built app points at the correct backend port.

## Install & run (local development)

1. Install dependencies for backend and frontend:

```powershell
cd backend
npm install

cd ../frontend
npm install
```

2. Start the backend (reads `backend/.env`):

```powershell
cd backend
node server.js
```

3. Start frontend (dev with HMR):

```powershell
cd frontend
npm run dev
```

Or preview the built app (the sync script will run automatically):

```powershell
cd frontend
npm run build
npm run preview
```

## Notes & troubleshooting
- If the frontend cannot reach the backend after building, ensure `backend/.env` contains `PORT` and that the backend is running. The sync script will copy the port into `frontend/.env` as `VITE_API_URL`.
- CORS: the backend is configured to accept `http://localhost:*` origins to support dev/preview servers. For production, set `FRONTEND_ORIGIN` to the exact origin and restrict CORS accordingly.
- Database: run `backend/schema.sql` to create the required tables. Update `DB_*` values in `backend/.env`.

## Testing the compare endpoint
- The compare endpoint is `GET /api/hardware-trees/compare?model=<model>&from_version=<from>&to_version=<to>` and returns a structured JSON diff with `removed_count`, `added_count`, `modified_count` and the full list of changed nodes.
- The diff algorithm matches hardware nodes by slot address with positional tie-breaking (handles B&R trees where IF1 placeholder nodes and real modules share the same address), then falls back to instance-label matching for unaddressed accessory nodes and IO variable entries.
- IO variable entries are matched by physical channel (the `AT %` address) so renames are reported as modifications rather than a remove + add. Each diff entry carries a `field` discriminator (`variable`, `address`, `article`, `location`) to drive targeted UI rendering.
- Verified against FA-MKII hardware trees containing ~3 000 nodes each (including ~2 900 IO mappings); the comparison completes in under a second.


## Importing configuration exports
Backend includes a CLI importer for B&R Automation Studio hardware configuration exports:

- Location: `backend/scripts/import-config.js`
- Purpose: read a tabular Excel export and insert a hardware tree + nodes into the database in a single transaction.

Usage:

```powershell
node scripts/import-config.js <excel-file> <model> <version> [tree-name] [--overwrite] [--version-to=X]
```

Examples:

```powershell
node scripts/import-config.js exports/example.xlsx "FA-MKIII" "V2.1.5.5" "MK-III"
node scripts/import-config.js exports/example.xlsx "FA-MKIII" "V2.1.5.5" --overwrite
node scripts/import-config.js exports/example.xlsx "FA-MKIII" "V2.1.5.0" "MK-III" --version-to=V2.1.9.9
```

Expected Excel format (sheet 1, no header row): three columns per row

- Column A: Article / order number (optional) ‚Äî stored as node `description`.
- Column B: Instance name + module description ‚Äî stored as node `name`.
- Column C: Position path string:
  - `$root` ‚Üí root node (top-level, parent_id = NULL)
  - `SL2.IF1.ST5` ‚Üí hierarchical path; parent is all-but-last segments
  - empty string `""` ‚Üí accessory / terminal block; becomes a child of the last positioned node

Behavior notes:
- Duplicate paths: the first row creates the node at that path; subsequent rows with the same path become children (e.g., memory keys, accessories).
- Missing intermediate paths are auto-created as lightweight placeholder nodes so the hierarchy remains intact.
- The importer runs inside a DB transaction and will roll back on error ‚Äî make a DB backup if needed before large imports.

Flags:
- `--overwrite` ‚Äî if a tree for the same `model`/`version` exists, delete its nodes and re-import.
- `--version-to=X` ‚Äî set the `version_to` field on the created tree record.

There is also a helper script `backend/scripts/export-template.js` and example command stubs under `backend/exports/` to generate example exports.

## Importing IO variable mappings

After importing a hardware tree you can attach IO variable mappings from a B&R Automation Studio IO export:

- Location: `backend/scripts/import-io.js`
- Purpose: parse IO variable declarations (`VAR AT` format) and insert them as children of the matching hardware nodes, keyed by instance name.

Usage:

```powershell
node scripts/import-io.js --file <excel-file> --tree-id <id>
node scripts/import-io.js --file <excel-file> --model <model> --version <version> [--overwrite]
```

Examples:

```powershell
node scripts/import-io.js --file exports/MKII-3_6_23_IO.xlsx --tree-id 3
node scripts/import-io.js --file exports/MKII-3_6_23_IO.xlsx --model "FA-MKIII" --version "V2.1.5.0" --overwrite
```

Each imported entry is stored with the PLC variable name as `name` and the physical channel in the form `AT %IX.DigitalInput02` as `description`. The compare tool uses the channel description as the stable match key so variable renames surface as `modified` entries rather than a remove + add.
