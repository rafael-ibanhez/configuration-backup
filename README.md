# Configuration Backup — Nilpeter

Small admin app to track hardware configurations (hardware trees) and backups for Nilpeter machines.

## Overview
- Backend: Node.js + Express API that stores hardware trees, nodes and backups in MySQL. Exposes REST endpoints under `/api/*` (equipment, slots, backups, hardware-trees, hardware-nodes).
- Frontend: SvelteKit (Vite) single-page app providing UI for equipment, hardware trees, compare views and import/export.
- Purpose: track hardware configurations across software versions, compare trees to detect added/removed/moved nodes, and manage backups.

## Features
- View and manage hardware trees and their nested nodes (slots).
- Create / copy / import hardware trees from Excel.
- Compare two software versions of a hardware tree to show added/removed/modified/moved nodes.
- Upload/download backups associated with equipment.
- Accessible UI improvements (keyboard-friendly modals, labeled inputs, image buttons).

## Repo structure
- `backend/` — Express API, DB access, routes.
  - `server.js` — app entry, CORS and routes.
  - `.env` — runtime settings (DB credentials, PORT, FRONTEND_ORIGIN).
  - `routes/` — route handlers for equipment, backups, hardware-trees, etc.
  - `db.js`, `schema.sql` — DB connection and schema.
- `frontend/` — SvelteKit app
  - `src/lib/api.js` — client API wrapper (reads `VITE_API_URL`).
  - `scripts/sync-backend-port.js` — copies backend port into `frontend/.env` as `VITE_API_URL` before build/dev.

## Environment & configuration

Backend `.env` (example):

```
DB_HOST=localhost
DB_PORT=3306
DB_USER=configUser
DB_PASSWORD=secret
DB_NAME=config_backup

PORT=3005
FRONTEND_ORIGIN=http://localhost:5173
```

Notes:
- `PORT` controls what port the backend listens on. The frontend client expects `VITE_API_URL` (set automatically by the sync script).
- `FRONTEND_ORIGIN` is used by the backend CORS policy. In development the server allows any `http://localhost:*` origin.

## Frontend sync script
The frontend includes `scripts/sync-backend-port.js` which reads `backend/.env` for `PORT` (or `BACKEND_PORT`) and writes `frontend/.env` containing `VITE_API_URL=http://localhost:<PORT>`.
The `package.json` `dev`, `build`, and `preview` scripts run this script first so the built app points at the correct backend port.

## Install & run (local development)

1. Install dependencies for backend and frontend:

```powershell
cd backend
npm install

cd ../frontend
npm install
```

2. Start the backend (reads `backend/.env`):

```powershell
cd backend
node server.js
```

3. Start frontend (dev with HMR):

```powershell
cd frontend
npm run dev
```

Or preview the built app (the sync script will run automatically):

```powershell
cd frontend
npm run build
npm run preview
```

## API base URL
- The frontend reads `VITE_API_URL` from `frontend/.env` generated by the sync script. Default fallback is `http://localhost:3000` inside `src/lib/api.js`.

## Notes & troubleshooting
- If the frontend cannot reach the backend after building, ensure `backend/.env` contains `PORT` and that the backend is running. The sync script will copy the port into `frontend/.env` as `VITE_API_URL`.
- CORS: the backend is configured to accept `http://localhost:*` origins to support dev/preview servers. For production, set `FRONTEND_ORIGIN` to the exact origin and restrict CORS accordingly.
- Database: run `backend/schema.sql` to create the required tables. Update `DB_*` values in `backend/.env`.

## Testing the compare endpoint
- The compare endpoint is `GET /api/hardware-trees/compare?model=<model>&from_version=<from>&to_version=<to>` and returns a JSON diff summary with `removed_count`, `added_count`, `modified_count` and lists of changed nodes.

## Importing configuration exports
Backend includes a CLI importer for B&R Automation Studio hardware configuration exports:

- Location: `backend/scripts/import-config.js`
- Purpose: read a tabular Excel export and insert a hardware tree + nodes into the database in a single transaction.

Usage:

```powershell
node scripts/import-config.js <excel-file> <model> <version> [tree-name] [--overwrite] [--version-to=X]
```

Examples:

```powershell
node scripts/import-config.js exports/example.xlsx "FA-MKIII" "V2.1.5.5" "MK-III"
node scripts/import-config.js exports/example.xlsx "FA-MKIII" "V2.1.5.5" --overwrite
node scripts/import-config.js exports/example.xlsx "FA-MKIII" "V2.1.5.0" "MK-III" --version-to=V2.1.9.9
```

Expected Excel format (sheet 1, no header row): three columns per row

- Column A: Article / order number (optional) — stored as node `description`.
- Column B: Instance name + module description — stored as node `name`.
- Column C: Position path string:
  - `$root` → root node (top-level, parent_id = NULL)
  - `SL2.IF1.ST5` → hierarchical path; parent is all-but-last segments
  - empty string `""` → accessory / terminal block; becomes a child of the last positioned node

Behavior notes:
- Duplicate paths: the first row creates the node at that path; subsequent rows with the same path become children (e.g., memory keys, accessories).
- Missing intermediate paths are auto-created as lightweight placeholder nodes so the hierarchy remains intact.
- The importer runs inside a DB transaction and will roll back on error — make a DB backup if needed before large imports.

Flags:
- `--overwrite` — if a tree for the same `model`/`version` exists, delete its nodes and re-import.
- `--version-to=X` — set the `version_to` field on the created tree record.

There is also a helper script `backend/scripts/export-template.js` and example command stubs under `backend/exports/` to generate example exports.
